01 — Why

설계 철학

AI 에이전트는 강력하지만, 규칙·경계·피드백이 없으면 빠르게 일관성을 잃습니다. 이 패키지는 그 통제를 두 가지 원칙으로 구조화합니다.

원칙 1 — 60줄 진입점 + 세부 분리

에이전트가 가장 먼저 읽는 AGENTS.md는 60줄 이하로 유지합니다. 1,000페이지 설명서는 읽히지 않습니다. 핵심 규칙만 진입점에 두고, 세부는 rules/·Skill/·MCP/로 분리해 필요할 때만 펴보게 합니다.

원칙 2 — 4대 운영 축

지침은 rules/ 아래 네 축으로 나뉩니다.

01_enforcement-system

강제 시스템

검증·테스트·커밋 강제. "성공은 조용히, 실패만 시끄럽게."

02_tool-boundaries

도구 경계

RAG·서브에이전트·MCP·팀·Windows 도구를 언제·어디까지 쓰는지 + 스킬/MCP 사용 가이드.

03_feedback-loop

피드백 루프

코딩 → 리뷰 → 규칙 보강 → 반복. 실수는 한 줄씩 누적.

04_garbage-collection

가비지 컬렉션

문서-코드 정합, 데드코드, 미사용 규칙 주기적 청소.

02 — Structure

패키지 구성

크게 지침 레이어(에이전트가 따르는 규칙)와 레퍼런스 레이어(필요할 때 펴보는 카탈로그)로 나뉩니다.

03 — Quick Start

빠른 시작

이 패키지를 환경에 배치하는 4단계입니다. (여기서 '배치'는 행동 규칙 적용의 전 단계 — 규칙 적용은 배치 후 AGENTS.md가 안내하며, 비-Claude AI는 PORTABILITY.md를 따른다.)

$ # 1. 저장소 클론
$ git clone <repo-url> Agent-Instructions

$ # 2. 작업할 프로젝트 루트에 배치 (또는 심볼릭 링크)
$ cp -r Agent-Instructions /path/to/your-project/

$ # 3. 에이전트에게 진입점을 읽히기
› "Agent-Instructions/AGENTS.md 를 읽고 그 규칙에 따라 작업해줘"

$ # 4. 스킬/MCP가 궁금하면 카탈로그 참조
› "Skill/README.md 에서 적합한 스킬을 찾아 적용"
⚠ 클론 직후 1가지

이 카탈로그는 특정 환경의 스냅샷입니다. 당신의 플러그인/MCP 구성이 다르면 유지보수 절차로 카탈로그를 갱신하세요.

04 — Install

설치 · 적용

배치 위치

  • 프로젝트 루트에 두면 해당 프로젝트 작업 시 에이전트가 진입점을 발견하기 쉽습니다.
  • 여러 프로젝트에서 공유하려면 공통 위치에 두고 각 프로젝트에서 참조하도록 안내합니다.

전제 조건

항목요구
Claude Code최신 버전 권장 (스킬·MCP·서브에이전트 지원)
플러그인카탈로그에 적힌 플러그인이 설치/활성(ON)이어야 해당 스킬이 실제로 동작
MCP 서버claude mcp add --scope user로 등록 (settings.json 직접 편집 금지)
WindowsBash $VAR 확장 안 됨, 따옴표 이슈 → 경로 직접 지정·Glob/Grep/Read 우선

에이전트가 AGENTS.md를 읽게 하는 법

Claude Code는 CLAUDE.md를 자동으로 읽습니다. AGENTS.md는 진입점으로 명시 참조하거나, 프로젝트 CLAUDE.md에서 링크해 두면 됩니다.

05 — Usage

사용법

작업 3단계 흐름

지침이 강제하는 기본 작업 흐름입니다.

단계내용
1. 디스커션핵심 의도 파악. 모호하면 먼저 질문(추측 금지). 실현 가능성·사이드이펙트 판단.
2. 요구사항 정리무엇을·어디서·어떻게·왜. 수정 범위·영향 파일·주의사항 정리 후 승인.
3. 실행도구로 수행. 예상과 다르면 즉시 멈추고 1단계로. 땜빵 금지, 근본 원인 해결.

스킬이 적용되면 먼저 호출하고 "Using [skill]"를 알립니다. 상황별 매핑은 rules/02에, 전체 목록은 Skill/README.md에 있습니다.

07 — Operate

규칙 운영 · 기여

팀 표준이므로 규칙은 살아있는 문서입니다. 에이전트가 같은 실수를 반복하면 규칙을 보강합니다.

규칙 추가 시 배치 기준

성격위치
나(사용자)에 대한 것 — 언어·스타일·습관~/.claude/CLAUDE.md
프로젝트 핵심 정보 — 구조·상태·워크플로우프로젝트 CLAUDE.md / 이 패키지
특정 주제 상세 — 트러블슈팅·도구 경계rules/
개인용·git 제외 — 로컬 경로·개인 설정CLAUDE.local.md

추가 전 체크

  • 이 규칙을 빼면 에이전트가 실제로 실수하는가? 아니면 넣지 않는다.
  • 린터/포매터가 잡아주는가? 그러면 넣지 않는다.
  • 해당 파일이 너무 길어지는가? 넘으면 rules/로 분리한다.

공개 저장소이므로 변경은 PR로 제안하고, 실패 로그(AGENTS.md 하단)에 재발 방지 한 줄을 누적합니다.

08 — Maintain

유지보수

가비지 컬렉션(rules/04) 원칙에 따라 주기적으로 점검합니다.

  • 문서-실제 정합 — 스킬/MCP를 추가·제거하면 Skill/·MCP/ 카탈로그와 README 숫자를 함께 갱신.
  • 버전 드리프트 — 플러그인 캐시 경로의 버전이 바뀌면 출처 표기 갱신.
  • 데드 항목 — 미사용 규칙·끊긴 링크·중복 경고 정리.

카탈로그 갱신 방법

플러그인 캐시(~/.claude/plugins/cache/)의 실제 SKILL.md를 근거로 작성합니다. 에이전트에게 "현재 스킬/MCP 구성을 다시 조사해 카탈로그를 갱신해줘"라고 요청하면 됩니다.

09 — Pre-deploy

공개 전 점검

공개 GitHub로 배포하기 전 반드시 확인합니다. 클릭해서 체크하세요.

  • 개인 절대경로 치환 — 스킬 md의 캐시 경로 C:\Users\<name>\… → ~/.claude/… 등 제네릭으로
  • 사용자명 노출 제거 — 경로·예시에 박힌 계정명/이메일
  • references 공개 적절성 — 비공개 프로젝트 도메인 지식이 들어있지 않은지
  • 내부 repo·링크 확인 — rules에 박힌 사내 저장소 URL의 공개 의도
  • 토큰·시크릿 부재 재확인 — 저장소 어디에도 토큰이 없는지
  • LICENSE · CONTRIBUTING 추가 — 공개 저장소 표준 파일
왜 중요한가

"그대로 패키징"하면 특정 환경의 개인 경로·계정명·내부 정보가 공개 저장소에 그대로 노출됩니다. 카탈로그가 당신의 설치 상태 스냅샷이라는 점도 받는 사람에게 명시해야 혼란이 없습니다.

10 — Limits

알려진 한계

  • 카탈로그 = 스냅샷 — 스킬 97·MCP 16은 작성 환경 기준. 다른 환경에선 목록이 다를 수 있음.
  • neon 스킬 원문 미vendoring — 원본 SKILL.md가 깨진 심볼릭 링크라 매니페스트 메타데이터 기반으로 작성(문서 자체는 완결). 원문은 원본 저장소 참조.
  • 캐시 경로 환경 종속 — 출처의 절대경로는 작성 환경 기준이라 그대로는 무의미(공개 전 치환 권장).
11 — FAQ

자주 묻는 질문

에이전트가 AGENTS.md를 안 읽어요.

Claude Code가 자동으로 읽는 것은 CLAUDE.md입니다. AGENTS.md는 진입점으로 명시 참조하거나 프로젝트 CLAUDE.md에서 링크해 두세요.

카탈로그의 스킬이 제 환경엔 없어요.

카탈로그는 작성 환경의 플러그인 설치 상태 스냅샷입니다. 해당 플러그인을 설치/활성(ON)하거나, 유지보수 절차로 카탈로그를 당신 구성에 맞게 갱신하세요.

스킬과 MCP의 차이가 뭔가요?

스킬은 Claude가 따르는 절차/방법론 패키지이고, MCP는 외부 도구·서비스를 연결하는 서버입니다. 카탈로그는 둘을 각각 Skill/·MCP/로 분리해 둡니다.

왜 README.md가 아니라 AGENTS.md가 진입점인가요?

README는 사람용 소개·사용법(이 문서), AGENTS.md는 AI가 따르는 행동 지침으로 역할이 분리됩니다. AI 도구 생태계의 공통 관례(AGENTS.md)를 따릅니다.
