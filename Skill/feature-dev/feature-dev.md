# feature-dev

> 코드베이스를 깊이 이해하고, 모호한 부분을 먼저 질문하고, 여러 아키텍처를 설계한 뒤 구현하는 7단계 가이드형 기능 개발 명령어.

## 📌 용도

`/feature-dev`은 새 기능 구현을 **체계적인 7단계 워크플로**로 안내하는 명령어다. 곧장 코딩에 뛰어들지 않고, 다음 순서로 진행한다.

1. 코드베이스를 깊이 이해 (서브에이전트로 탐색)
2. 모든 모호함·엣지케이스·미명세 동작을 **먼저 질문**
3. 트레이드오프가 다른 **여러 아키텍처를 설계**해 비교
4. 사용자 승인 후 구현
5. 다각도 품질 리뷰

핵심 원칙:
- **명확화 질문 우선** — 가정 대신 구체적 질문, 답을 받기 전엔 구현 진행 금지
- **행동 전 이해** — 기존 코드 패턴부터 읽고 파악
- **에이전트가 식별한 파일을 직접 읽기** — 에이전트가 반환한 핵심 파일 목록을 메인에서 Read해 상세 컨텍스트 확보
- **단순하고 우아하게** — 읽기 쉽고 유지보수 가능한 아키텍처
- **TodoWrite로 전 과정 추적**

## 🎯 트리거 조건

- **언제 호출되는가**
  - 사용자가 `/feature-dev` 또는 `/feature-dev:feature-dev`을 입력할 때 (선택적으로 기능 설명을 인자로)
  - 새 기능을 코드베이스 이해 → 설계 → 구현 순으로 신중하게 진행하고 싶을 때
  - 기존 패턴·아키텍처와 잘 맞물리는 구현이 중요한 중대형 기능일 때

- **언제 쓰지 않는가**
  - 한 줄 수정·오타 수정 같은 사소한 변경 (7단계 오버헤드가 과함)
  - 성공 기준이 자동 검증 가능한 반복형 작업 → `/ralph-loop`가 적합
  - 이미 설계가 확정되어 곧장 구현만 하면 되는 경우
  - 코드 리뷰만 필요한 경우 → `/review-pr`가 적합

## 💻 사용 예시 / 명령어

기능 설명 없이 호출 (Discovery 단계에서 물어봄):
```
/feature-dev
```

기능 설명을 인자로 (`argument-hint: Optional feature description`):
```
/feature-dev 사용자 프로필에 아바타 업로드 기능 추가
/feature-dev:feature-dev 결제 모듈에 환불 처리 흐름 구현
```

호출하면 명령은 7단계를 순서대로 진행하며, 각 단계의 게이트(질문 답변, 접근법 선택, 구현 승인)에서 **사용자 입력을 기다린다**.

## 🛠️ 제공 도구 / 주요 파라미터

**7단계 워크플로:**

- **Phase 1 — Discovery**: 무엇을 만들지 이해. 전체 단계 todo 생성, 기능이 불명확하면 문제·동작·제약을 질문, 이해 내용 확인.
- **Phase 2 — Codebase Exploration**: **code-explorer 에이전트 2~3개를 병렬 실행**, 각자 다른 측면(유사 기능, 고수준 이해, 아키텍처, UX)을 담당하고 읽어야 할 핵심 파일 5~10개 반환. 에이전트 종료 후 그 파일들을 **모두 Read**해 깊은 이해 구축, 발견 사항 요약.
- **Phase 3 — Clarifying Questions**: (가장 중요, 절대 생략 금지) 엣지케이스·에러 처리·통합 지점·범위 경계·설계 선호·하위호환성·성능 등 미명세 항목을 정리해 **명확한 목록으로 질문**하고 답을 기다림. "알아서 해줘"라고 하면 추천안을 제시하고 명시적 확인을 받음.
- **Phase 4 — Architecture Design**: **code-architect 에이전트 2~3개 병렬 실행**, 각각 다른 초점(최소 변경 / 클린 아키텍처 / 실용적 균형). 접근법을 비교하고 **추천안과 근거**를 제시한 뒤 어느 접근을 선호하는지 질문.
- **Phase 5 — Implementation**: (사용자 승인 없이 시작 금지) 명시적 승인 후, 관련 파일을 모두 읽고 선택된 아키텍처대로 구현, 코드베이스 컨벤션을 엄격히 준수, todo 갱신.
- **Phase 6 — Quality Review**: **code-reviewer 에이전트 3개 병렬 실행**, 각각 단순성/DRY/우아함 · 버그/기능 정확성 · 프로젝트 컨벤션/추상화에 초점. 최고 심각도 이슈를 모아 **사용자에게 제시하고 처리 방식(지금 수정/나중/그대로 진행) 질문**.
- **Phase 7 — Summary**: 모든 todo 완료 처리, 만든 것·핵심 결정·수정 파일·다음 단계 요약.

**사용하는 서브에이전트:** `code-explorer`(코드베이스 탐색·추적), `code-architect`(아키텍처 설계), `code-reviewer`(품질 리뷰). 모두 병렬 실행으로 메인 컨텍스트를 절약하고 다양한 관점을 확보한다.

**주요 동작 특성:** 단계마다 사용자 승인 게이트가 있어 잘못된 방향으로 진행되지 않으며, 에이전트가 식별한 파일을 메인에서 직접 읽어 컨텍스트를 축적하는 패턴이 반복된다.
