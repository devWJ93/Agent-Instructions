# verification-before-completion

> 작업이 완료·수정·통과됐다고 주장하기 전에, 반드시 검증 명령을 실행하고 그 출력을 확인하라는 스킬 — 주장보다 증거가 항상 먼저.

## 📌 용도
검증 없이 작업 완료를 주장하는 것은 효율이 아니라 부정직이다. 핵심 원칙은 "주장보다 증거가 항상 먼저"다. 철의 법칙: **신선한 검증 증거 없이는 완료 주장 금지** — 이 메시지에서 검증 명령을 실행하지 않았다면 통과한다고 주장할 수 없다. 성공/완료를 시사하는 어떤 표현(정확한 문구·동의어·함의 포함)이든 적용된다.

## 🎯 트리거 조건
- **언제 호출되는가**: 작업이 완료/수정/통과됐다고 주장하기 직전, 커밋·PR 생성 전. 성공·완료·만족을 표현하기 전 항상.
- **언제 쓰지 않는가**: (별도 제외 조건 없음. 모든 완료성 주장에 적용.)
- **스킬 유형**: Rigid — "규칙의 문구를 어기는 것은 정신을 어기는 것", 비협상 게이트 함수 강제.

## 💻 사용 예시 / 호출 방법
게이트 함수(어떤 상태 주장·만족 표현 전):
1. **IDENTIFY** — 이 주장을 증명하는 명령은?
2. **RUN** — 전체 명령을 신선하게·완전히 실행
3. **READ** — 전체 출력 읽고, exit code 확인, 실패 개수 세기
4. **VERIFY** — 출력이 주장을 확인하나? NO면 실제 상태를 증거와 함께 진술, YES면 주장을 증거와 함께
5. **ONLY THEN** — 주장한다. (어느 단계든 건너뛰면 검증이 아니라 거짓말.)

핵심 패턴:
- 테스트: ✅ [명령 실행][34/34 통과 확인]"모든 테스트 통과" / ❌ "이제 통과할 것"
- 회귀 테스트(TDD Red-Green): ✅ 작성 → 실행(통과) → 수정 되돌림 → 실행(반드시 실패) → 복원 → 실행(통과) / ❌ "회귀 테스트 작성함"(red-green 검증 없이)
- 빌드: ✅ [빌드 실행][exit 0]"빌드 통과" / ❌ "린터 통과"(린터는 컴파일 검사 안 함)
- 요구사항: ✅ 계획 재독 → 체크리스트 → 각 검증 → 갭/완료 보고 / ❌ "테스트 통과, 단계 완료"
- 에이전트 위임: ✅ 에이전트 성공 보고 → VCS diff 확인 → 변경 검증 → 실제 상태 보고 / ❌ 에이전트 보고 신뢰

## 🛠️ 제공 내용 / 핵심 절차
- **흔한 실패 표**: "테스트 통과"는 테스트 명령 출력 0 실패 필요(이전 실행·"통과할 것"은 불충분), "린터 클린", "빌드 성공"(exit 0), "버그 수정"(원래 증상 테스트), "에이전트 완료"(VCS diff), "요구사항 충족"(한 줄씩 체크리스트) 등.
- **Red Flags(STOP)**: "should/probably/seems to" 사용, 검증 전 만족 표현("Great!"/"Perfect!"/"Done!"), 검증 없이 커밋/푸시/PR, 에이전트 성공 보고 신뢰, 부분 검증 의존, "이번 한 번만", 피곤해서 끝내고 싶음, 검증 없이 성공을 함의하는 모든 표현.
- **합리화 방지표**: "이제 될 거야"→실행하라, "확신함"→확신≠증거, "린터 통과"→린터≠컴파일러, "에이전트가 성공이라 함"→독립 검증, "피곤"→피로≠변명, "부분 확인이면 충분"→부분은 아무것도 증명 못 함, "다른 단어니까 규칙 안 적용"→문구보다 정신.
- **연계**: `systematic-debugging`(수정 검증), `test-driven-development`(회귀 테스트 red-green).
- "명령을 실행하라. 출력을 읽어라. 그다음 결과를 주장하라. 비협상이다."
