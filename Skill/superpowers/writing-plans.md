# writing-plans

> 스펙·요구사항을 받아, 코드를 건드리기 전에 잘게 쪼갠 태스크 단위의 완전한 구현 계획을 작성하는 스킬.

## 📌 용도
우리 코드베이스에 대한 컨텍스트가 전혀 없고 취향도 의심스러운 엔지니어를 가정하고, 포괄적인 구현 계획을 작성한다. 각 태스크마다 어떤 파일을 건드릴지, 코드, 테스트, 확인할 문서, 테스트 방법까지 전부 문서화하고, 전체 계획을 잘게 쪼갠(bite-sized) 태스크로 제공한다. DRY·YAGNI·TDD·잦은 커밋을 따른다. 엔지니어가 숙련됐지만 우리 툴셋·도메인과 좋은 테스트 설계는 거의 모른다고 가정한다. 시작 시 "writing-plans 스킬을 사용해 구현 계획을 만듭니다"라고 알리고, 계획은 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`에 저장한다.

## 🎯 트리거 조건
- **언제 호출되는가**: 멀티스텝 작업의 스펙/요구사항이 있고, 코드를 건드리기 전. brainstorming의 종착 단계로 호출됨.
- **언제 쓰지 않는가**: 스펙이 여러 독립 서브시스템을 포함하면 먼저 plan을 분리(서브시스템당 하나, 각각 독립적으로 작동·테스트 가능)하도록 제안. 아직 스펙이 없으면 brainstorming 먼저.
- **스킬 유형**: Rigid에 가까움 — 계획 헤더, 태스크 구조, "No Placeholders" 규칙을 엄격히 강제.

## 💻 사용 예시 / 호출 방법
- **스코프 체크**: 여러 독립 서브시스템이면 별도 계획으로 분리 제안.
- **파일 구조**: 태스크 정의 전 어떤 파일을 생성/수정하고 각자 무엇을 책임지는지 매핑. 단위는 명확한 경계·인터페이스, 파일당 하나의 책임. 같이 변하는 파일은 같이 두기(기술 계층이 아니라 책임으로 분리).
- **잘게 쪼갠 단계(2~5분)**: "실패 테스트 작성" / "실패하는지 실행" / "통과시킬 최소 코드 구현" / "테스트 실행해 통과 확인" / "커밋" 각각이 한 단계.
- **계획 문서 헤더(필수)**: 제목 + "For agentic workers: REQUIRED SUB-SKILL: subagent-driven-development(권장) 또는 executing-plans, 단계는 체크박스 `- [ ]` 문법" + Goal/Architecture/Tech Stack.
- **태스크 구조**: `### Task N` 아래 Files(Create/Modify `path:line`/Test) + 체크박스 단계들(실패 테스트 코드 → 실행·실패 예상 → 최소 구현 코드 → 실행·통과 예상 → 커밋 명령). 코드 단계는 실제 코드 블록 포함.

## 🛠️ 제공 내용 / 핵심 절차
- **No Placeholders(계획 실패 — 절대 금지)**: "TBD/TODO/나중에 구현", "적절한 에러 처리 추가"/"검증 추가"/"엣지케이스 처리", "위 내용에 대한 테스트 작성"(실제 테스트 코드 없이), "Task N과 유사"(코드를 반복할 것 — 순서대로 안 읽을 수 있음), 코드 블록 없이 무엇을 할지만 설명, 어느 태스크에도 정의 안 된 타입·함수 참조.
- **기억할 것**: 항상 정확한 파일 경로, 모든 단계에 완전한 코드(코드 변경 단계면 코드 표시), 정확한 명령·기대 출력, DRY/YAGNI/TDD/잦은 커밋.
- **셀프 리뷰(직접 실행, 서브에이전트 아님)**: ①스펙 커버리지(각 요구사항에 대응 태스크가 있나, 갭 나열) ②placeholder 스캔 ③타입 일관성(앞 태스크에서 정의한 타입·시그니처·이름이 뒤 태스크와 일치하나 — `clearLayers()` vs `clearFullLayers()`는 버그). 인라인 수정, 갭 있으면 태스크 추가.
- **실행 핸드오프**: 저장 후 실행 선택 제시 — 1) Subagent-Driven(권장, 태스크당 새 서브에이전트·태스크 간 리뷰) → 필수 서브스킬 `subagent-driven-development`, 2) Inline Execution(이 세션 배치·체크포인트) → 필수 서브스킬 `executing-plans`.
