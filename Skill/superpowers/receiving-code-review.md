# receiving-code-review

> 코드 리뷰 피드백을 받을 때, 형식적 동의나 맹목적 구현 대신 기술적 검증과 엄밀함으로 평가·대응하는 스킬.

## 📌 용도
코드 리뷰는 감정적 퍼포먼스가 아니라 기술적 평가다. 핵심 원칙은 "구현 전 검증, 가정 전 질문, 사회적 편안함보다 기술적 정확성"이다. 피드백을 받으면 먼저 전체를 반응 없이 읽고, 요구사항을 자기 말로 재진술(또는 질문)하고, 코드베이스 현실과 대조 검증하고, 이 코드베이스에 기술적으로 타당한지 평가한 뒤, 기술적 인정 또는 근거 있는 반박으로 응답하고, 한 항목씩 테스트하며 구현한다. "당신 말이 전적으로 옳습니다" 같은 형식적 동의나 감사 표현은 금지된다.

## 🎯 트리거 조건
- **언제 호출되는가**: 코드 리뷰 피드백을 받았을 때, 제안을 구현하기 전, 특히 피드백이 불명확하거나 기술적으로 의심스러울 때.
- **언제 쓰지 않는가**: (별도 제외 조건은 없음. 모든 리뷰 수신 상황에 적용 권장.)
- **스킬 유형**: Rigid — 형식적 동의·감사 표현 금지, 검증 없는 구현 금지 등 명확한 규칙 강제.

## 💻 사용 예시 / 호출 방법
응답 패턴: READ(반응 없이 전체 읽기) → UNDERSTAND(자기 말로 재진술/질문) → VERIFY(코드베이스 대조) → EVALUATE(이 스택에 타당한가) → RESPOND(기술적 인정 or 근거 있는 반박) → IMPLEMENT(한 항목씩, 각각 테스트).

불명확한 피드백 처리: 한 항목이라도 불명확하면 **아무것도 구현하지 말고 멈추고** 불명확한 항목을 질문. (항목들이 연관될 수 있어 부분 이해 = 잘못된 구현.) 예: "1~6 고쳐" 중 1,2,3,6 이해·4,5 불명확이면 → "1,2,3,6은 이해, 4와 5는 구현 전 명확화 필요."

## 🛠️ 제공 내용 / 핵심 절차
- **금지 응답**: "You're absolutely right!"(CLAUDE.md 위반), "Great point!"/"Excellent feedback!"(형식적), "Let me implement that now"(검증 전). → 대신 요구사항 재진술/질문/근거 있는 반박/그냥 작업 시작.
- **출처별 처리**: 파트너 발 = 신뢰하되 스코프 불명확하면 질문. 외부 리뷰어 발 = ①이 코드베이스에 맞나 ②기존 기능 깨나 ③현재 구현 이유 ④모든 플랫폼/버전 동작 ⑤전체 컨텍스트 이해하나 점검. 검증 불가하면 "X 없이 검증 불가, [조사/질문/진행]할까요?"라고 명시. 파트너의 과거 결정과 충돌하면 먼저 논의.
- **YAGNI 체크**: 리뷰어가 "제대로 구현하라" 하면 grep으로 실제 사용 확인 — 미사용이면 "이 엔드포인트 호출 안 됨. 제거(YAGNI)?".
- **구현 순서**: 불명확 항목 먼저 명확화 → 블로킹(깨짐/보안) → 단순 수정(오타/임포트) → 복잡 수정(리팩토링/로직). 각각 개별 테스트, 회귀 확인.
- **반박 시점**: 기존 기능 깨짐, 리뷰어 컨텍스트 부족, YAGNI 위반, 스택에 부적합, 레거시/호환 이유, 파트너 아키텍처 결정과 충돌. 기술적 근거로, 방어적이지 않게.
- **올바른 피드백 인정**: "Fixed. [변경 설명]" / "Good catch - [이슈]. [위치]에서 수정." 또는 그냥 코드로. **모든 감사 표현 금지**("Thanks for...") — 막 쓰려 하면 삭제하고 수정 내용을 진술.
- **반박이 틀렸을 때**: "당신이 옳았습니다 - [X] 확인했고 [Y]합니다. 구현 중." 사실만 간결히, 긴 사과·변명 금지.
- **GitHub**: 인라인 리뷰 코멘트는 최상위 PR 코멘트가 아니라 해당 스레드에 답글(`gh api .../comments/{id}/replies`).
- **불편하면 신호**: "Strange things are afoot at the Circle K".
