# brainstorming

> 모든 창작 작업(기능/컴포넌트/동작 추가·변경) 전에 반드시 거쳐야 하는, 아이디어를 검증된 설계·스펙으로 발전시키는 대화형 디스커션 스킬.

## 📌 용도
사용자의 막연한 아이디어를 자연스러운 협업 대화를 통해 완성된 설계와 스펙으로 만든다. 먼저 현재 프로젝트 컨텍스트(파일, 문서, 최근 커밋)를 파악한 뒤, 질문을 한 번에 하나씩 던지며 목적·제약·성공 기준을 명확히 한다. 무엇을 만들지 이해되면 설계를 섹션별로 제시하고 각 섹션마다 사용자 승인을 받는다. 최종적으로 설계 문서를 `docs/superpowers/specs/`에 저장·커밋하고, 사용자 리뷰를 거친 뒤 `writing-plans` 스킬로 넘긴다. 어떤 단순한 프로젝트라도 예외 없이 이 과정을 거친다.

## 🎯 트리거 조건
- **언제 호출되는가**: 기능 생성, 컴포넌트 구축, 기능성 추가, 동작 수정 등 모든 창작 작업을 시작하기 전. "X를 만들자", "Y 기능 추가해줘" 같은 요청. EnterPlanMode 직전에 아직 브레인스토밍을 안 했다면 반드시 호출.
- **언제 쓰지 않는가**: 이미 설계 승인이 끝나 구현 단계로 넘어간 경우. 단순 조회/질문은 해당하지 않음(단, 작업으로 이어질 가능성이 있으면 적용).
- **스킬 유형**: Rigid(엄격) — HARD-GATE가 존재한다. 설계 제시·승인 전에는 어떤 구현 스킬 호출/코드 작성/스캐폴딩도 금지.

## 💻 사용 예시 / 호출 방법
워크플로우(체크리스트 9단계):
1. 프로젝트 컨텍스트 탐색 (파일/문서/최근 커밋)
2. (시각적 질문이 예상되면) Visual Companion 제안 — 반드시 단독 메시지로
3. 명확화 질문 — 한 번에 하나씩, 목적/제약/성공기준 파악
4. 2~3개 접근법 제안 (트레이드오프 + 추천안)
5. 설계 제시 — 복잡도에 맞춰 섹션별, 각 섹션마다 승인
6. 설계 문서 작성 — `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`에 저장·커밋
7. 스펙 셀프 리뷰 — placeholder/모순/모호함/스코프 인라인 점검
8. 사용자 스펙 리뷰 게이트
9. 구현으로 전환 — `writing-plans` 스킬 호출 (유일한 다음 스킬)

## 🛠️ 제공 내용 / 핵심 절차
- **HARD-GATE**: 설계 승인 전 구현 행위 전면 금지. "이건 너무 단순해서 설계 불필요"는 안티패턴 — 단순 프로젝트일수록 미검증 가정이 가장 큰 낭비를 부른다.
- **스코프 체크**: 여러 독립 서브시스템을 포함하면 즉시 플래그하고 sub-project로 분해. 각 sub-project가 spec → plan → 구현 사이클을 따로 가진다.
- **핵심 원칙**: 한 번에 한 질문 / 객관식 선호 / YAGNI 철저 / 항상 2~3개 대안 / 점진적 검증 / 유연하게 되돌아가기.
- **격리·명료성 설계**: 시스템을 하나의 명확한 목적을 가진 작은 단위로 분해, 잘 정의된 인터페이스로 통신, 독립 테스트 가능하게.
- **스펙 셀프 리뷰**: ①placeholder 스캔 ②내부 일관성 ③스코프 ④모호성 — 인라인 수정.
- **참조 파일**: `skills/brainstorming/visual-companion.md` (Visual Companion 동의 시 읽는 상세 가이드).
- **종착점**: 반드시 `writing-plans` 호출. frontend-design/mcp-builder 등 다른 구현 스킬 호출 금지.
