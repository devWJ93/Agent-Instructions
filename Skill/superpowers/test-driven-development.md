# test-driven-development

> 어떤 기능·버그픽스든 구현 코드 작성 전에 실패하는 테스트를 먼저 쓰고, 실패를 직접 확인한 뒤 최소 코드로 통과시키는 TDD 스킬.

## 📌 용도
테스트를 먼저 쓰고, 실패를 지켜보고, 통과시킬 최소 코드를 쓴다. 핵심 원칙은 "테스트가 실패하는 걸 직접 보지 않았다면, 그 테스트가 옳은 것을 검증하는지 알 수 없다"이다. 철의 법칙: **실패하는 테스트 없이는 프로덕션 코드 금지**. 테스트보다 코드를 먼저 썼다면 삭제하고 처음부터 다시("reference로 보관"·"적응"·"보기"도 금지, 삭제는 삭제). RED-GREEN-REFACTOR 사이클을 따른다.

## 🎯 트리거 조건
- **언제 호출되는가(항상)**: 새 기능, 버그 수정, 리팩토링, 동작 변경 — 구현 코드 작성 전.
- **언제 쓰지 않는가(파트너에게 물을 예외)**: 버리는 프로토타입, 생성된 코드, 설정 파일. "이번 한 번만 건너뛰자"는 생각은 합리화이니 멈출 것.
- **스킬 유형**: Rigid — "규칙의 문구를 어기는 것은 정신을 어기는 것"이며 철의 법칙을 강제.

## 💻 사용 예시 / 호출 방법
RED-GREEN-REFACTOR:
1. **RED 실패 테스트 작성** — 하나의 동작만, 명확한 이름, 실제 코드(불가피하지 않으면 mock 금지). 무엇이 일어나야 하는지 보이는 최소 테스트 1개.
2. **Verify RED(필수, 절대 생략 금지)** — `npm test path/...`로 실행해 ①에러가 아닌 fail ②기대한 실패 메시지 ③기능 부재로 인한 실패(오타 아님) 확인. 통과하면 기존 동작을 테스트한 것 → 테스트 수정. 에러면 고쳐서 올바르게 실패할 때까지.
3. **GREEN 최소 코드** — 테스트를 통과시킬 가장 단순한 코드. 기능 추가·다른 코드 리팩토링·"개선" 금지(YAGNI).
4. **Verify GREEN(필수)** — 테스트 통과 + 다른 테스트도 통과 + 출력 깨끗(에러/경고 없음). 실패하면 코드를 고침(테스트 아님).
5. **REFACTOR** — green 이후에만 중복 제거·이름 개선·헬퍼 추출, 테스트는 green 유지, 동작 추가 금지.
6. 반복.

## 🛠️ 제공 내용 / 핵심 절차
- **좋은 테스트**: 최소(한 가지, 이름에 "and" 있으면 분리), 명확(동작을 설명하는 이름), 의도 표현(원하는 API 시연).
- **순서가 중요한 이유**: 코드 후 작성한 테스트는 즉시 통과 → 아무것도 증명 못 함(잘못된 것 테스트, 구현을 테스트, 엣지케이스 누락, 버그 잡는 걸 못 봄). 테스트 우선이라야 엣지케이스를 구현 전에 발견.
- **합리화 반박표**: "너무 단순", "나중에 테스트", "이미 수동 테스트", "X시간 삭제는 낭비(매몰비용)", "reference로 보관", "TDD는 교조적", "테스트 어려움 = 설계 불명확(테스트 어려우면 쓰기도 어려움)" 등.
- **Red Flags(삭제·재시작)**: 테스트 전 코드, 구현 후 테스트, 즉시 통과, 왜 실패했는지 설명 불가, 나중에 테스트 추가, "이번만", "수동 테스트했음", "정신이 중요", "reference 보관/기존 코드 적응".
- **검증 체크리스트**: 모든 새 함수에 테스트 / 구현 전 실패 확인 / 기대한 이유로 실패 / 최소 코드로 통과 / 전체 통과 / 출력 깨끗 / 실제 코드 사용 / 엣지·에러 커버. 다 체크 못 하면 TDD를 건너뛴 것 → 재시작.
- **막혔을 때**: 테스트법 모름 → 원하는 API/단언 먼저 작성. 테스트 너무 복잡 → 설계 복잡, 인터페이스 단순화. 전부 mock 필요 → 결합 과다, DI 사용. 셋업 거대 → 헬퍼 추출.
- **참조**: `@testing-anti-patterns.md`(mock·테스트 유틸 추가 시 — mock 동작 테스트, 프로덕션에 테스트 전용 메서드 추가, 의존성 이해 없는 mocking 회피).
- **디버깅 연계**: 버그 발견 시 재현하는 실패 테스트 먼저 작성 후 TDD 사이클. 테스트 없이 버그 수정 금지.
