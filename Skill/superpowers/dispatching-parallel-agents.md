# dispatching-parallel-agents

> 서로 의존하지 않는 2개 이상의 독립 작업을, 격리된 컨텍스트를 가진 전용 에이전트에게 병렬로 위임하는 스킬.

## 📌 용도
여러 개의 서로 무관한 실패(다른 테스트 파일, 다른 서브시스템, 다른 버그)를 순차적으로 조사하면 시간이 낭비된다. 이 스킬은 각 독립 문제 영역마다 하나의 에이전트를 디스패치하여 동시에 작업하게 한다. 각 에이전트는 정밀하게 작성된 지시와 컨텍스트만 받으며, 절대 메인 세션의 히스토리를 상속하지 않는다. 이로써 에이전트는 자기 작업에 집중하고, 메인 컨텍스트는 조율 작업용으로 보존된다. 핵심 원칙은 "독립 문제 도메인당 에이전트 1개, 동시 실행"이다.

## 🎯 트리거 조건
- **언제 호출되는가**: 3개 이상 테스트 파일이 서로 다른 근본 원인으로 실패할 때, 여러 서브시스템이 독립적으로 깨졌을 때, 각 문제를 다른 문제의 컨텍스트 없이 이해할 수 있고 조사 간 공유 상태가 없을 때.
- **언제 쓰지 않는가**: 실패들이 서로 연관(하나 고치면 다른 것도 고쳐질 수 있음), 전체 시스템 상태를 봐야 이해 가능, 무엇이 깨졌는지 모르는 탐색적 디버깅, 에이전트들이 같은 파일/자원을 건드려 충돌할 때.
- **스킬 유형**: Flexible — 패턴/기법형. 상황 판단(독립성·병렬 가능성)에 맞춰 적용.

## 💻 사용 예시 / 호출 방법
패턴 4단계:
1. **독립 도메인 식별** — 실패를 깨진 대상별로 묶기 (파일 A: 도구 승인 흐름, 파일 B: 배치 완료, 파일 C: abort).
2. **집중된 에이전트 태스크 생성** — 구체적 스코프(파일 1개), 명확한 목표(이 테스트 통과), 제약(다른 코드 변경 금지), 기대 출력(발견·수정 요약).
3. **병렬 디스패치** — `Task("Fix agent-tool-abort.test.ts failures")` 등 동시 실행.
4. **리뷰·통합** — 각 요약 읽기, 충돌 확인, 전체 테스트 스위트 실행, 통합.

## 🛠️ 제공 내용 / 핵심 절차
- **좋은 에이전트 프롬프트 3요소**: ①집중(하나의 명확한 문제 도메인) ②자기완결(문제 이해에 필요한 모든 컨텍스트) ③구체적 출력(무엇을 반환할지).
- **흔한 실수**:
  - ❌ 너무 광범위 "모든 테스트 고쳐" → ✅ "agent-tool-abort.test.ts 고쳐"
  - ❌ 컨텍스트 없음 → ✅ 에러 메시지·테스트 이름 붙여넣기
  - ❌ 제약 없음(전부 리팩토링할 위험) → ✅ "프로덕션 코드 변경 금지"
  - ❌ 모호한 출력 "고쳐" → ✅ "근본 원인·변경 요약 반환"
- **검증**: 에이전트 복귀 후 각 요약 리뷰 → 충돌 확인(같은 코드 수정했나) → 전체 스위트 실행 → 스폿 체크(에이전트는 체계적 오류 가능).
- **핵심 이점**: 병렬화, 집중, 독립성, 속도(3문제를 1문제 시간에).
