# 01. 자동 강제 시스템 (Enforcement System)

규칙은 "지키자고 다짐"하는 게 아니라 **기계가 강제**할 때만 지켜진다. 이 문서는 무엇을·언제 강제하고, 실패를 어떻게 드러내는지 규정한다.

> **현재 워크스페이스 상태**: `Agent/` 폴더는 git 저장소도, 단일 언어 코드베이스도 아니다. 따라서 아래 강제 지점은 지금은 **에이전트의 수동 규율**로 작동한다. 실제 프로젝트에서는 해당 절을 자동화 훅으로 연결한다. "향후 연결 지점" 표시를 참고한다.

## 1. 검증 게이트 (가장 중요)

코드·설정을 수정했으면 **완료를 선언하기 전에** 반드시 실행해서 확인한다.

- 코드를 건드렸으면 `npm test` · `pytest` · `cargo test` 등 프로젝트가 쓰는 테스트를 돌린다.
- 테스트 셋업이 없으면 최소한 빌드/컴파일이 되는지 확인한다.
- 사용자가 "끝"·"완료"·"다 됐어" 신호를 보내기 **전에** 선제적으로 돌린다.
- 통과하면 결과를 보고하고, 실패하면 고친 뒤 다시 돌린다.
- **검증 없이 커밋 금지.** 검증은 협상 대상이 아니다.

→ 향후 연결 지점: pre-commit 훅에서 테스트/빌드 실패 시 커밋 차단.

## 2. 린터·포매터 경계

- 린터/포매터가 자동으로 잡아 주는 항목은 규칙 문서에 넣지 않는다(중복).
- 규칙 문서에는 "도구가 못 잡는, 사람이 자주 틀리는 것"만 남긴다.
- 린터가 빨간불이면 에이전트가 스스로 수정하고, 사람 개입을 요구하지 않는다(자동 교정 루프).

→ 향후 연결 지점: 프로젝트별 린터 설정(.eslintrc, ruff 등) + 저장 시 자동 포맷.

## 3. 목표 주도 실행 (Goal-Driven)

모호한 기준("make it work")은 강제할 수 없다. 작업을 **검증 가능한 목표**로 바꾼다.

- "검증 추가" → "잘못된 입력 테스트를 작성하고 통과시킨다."
- "버그 수정" → "재현 테스트를 작성하고 통과시킨다."
- "X 리팩토링" → "전후 테스트가 모두 통과함을 확인한다."

다단계 작업은 짧은 계획을 명시한다.

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
```

## 4. 커밋 강제 (Semantic Commits)

- 한 논리적 변경이 끝나면 시키지 않아도 커밋한다.
- 테스트: "이 커밋을 한 문장으로 설명 가능한가?" 안 되면 변경이 섞인 것이니 분리한다.
- 좋은 예: "auth 미들웨어 추가". 나쁜 예: "auth 추가하고 UI 고치고 버그 수정"(3개로 분리).
- 코드 변경에 딸린 문서(README·API 문서 등) 업데이트도 같은 커밋의 일부다.
- 커밋 메시지는 한글로 작성한다.

## 5. 성공은 조용히, 실패만 시끄럽게

- 4,000줄이 통과한 결과를 장황하게 보여주지 않는다. 통과는 한 줄로 요약한다.
- 대신 **실패·경고·예외**는 전체 메시지와 스택 트레이스를 그대로 드러낸다.
- 추정 로그가 아니라 실제 출력에 근거해 판단한다(→ `03_feedback-loop.md`의 디버깅 규율과 연결).
