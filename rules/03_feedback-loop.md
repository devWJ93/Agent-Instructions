# 03. 피드백 루프 (Feedback Loop)

에이전트가 직접 코딩 → 리뷰 → **규칙 보강** → 반복. 같은 실수를 두 번 하지 않도록 규칙을 살아 있게 유지한다.

## 1. 리뷰

- 논리적 단위를 완성하면 스스로 리뷰하거나, code-reviewer 같은 전용 서브에이전트에 리뷰를 맡긴다.
- 리뷰는 "잘 했나"가 아니라 "요구사항을 충족했나 · 버그/사이드이펙트는 없나"를 본다.
- 리뷰 피드백을 받으면 맹목적으로 수용하지 않는다. 기술적으로 검증한 뒤 반영한다.

## 2. 실패에서 규칙으로

- 여러 번 시도 끝에 찾은 해결책은 **즉시** 기록한다. 프로젝트 CLAUDE.md 또는 트러블슈팅 문서에 남긴다.
- 에이전트가 같은 실수를 반복하면, 재발 방지 한 줄을 `AGENTS.md`의 "실패 로그"에 추가한다.
- 컨텍스트 파일은 1,000페이지 설명서가 아니다. 한 줄씩 누적하되 60줄을 넘기면 세부를 별도 문서로 분리한다.

## 3. 디버깅 규율 (추측 금지)

- 실제 에러 메시지와 스택 트레이스를 끝까지 읽는다. 기억으로 패턴 매칭하지 않는다.
- 원인을 확인하기 전에 "흔한 수정"을 적용하지 않는다.
- 불명확하면 print/log를 추가해 상태를 확인한 뒤 고친다.
- 이렇게 해야 한 줄 버그가 세 파일 리팩토링으로 번지지 않는다.

## 4. 규칙 추가 기준

새 규칙을 넣기 전에 세 가지를 자문한다.

- 이걸 빼면 에이전트가 실제로 실수하는가? 아니면 넣지 않는다.
- 린터/포매터가 잡아 주는가? 그러면 넣지 않는다(→ `01_enforcement-system.md`).
- 해당 문서가 너무 길어지는가? 60줄/150줄을 넘기면 세부를 분리한다.

## 5. 규칙 배치 기준

규칙은 올바른 위치에 둬야 작동한다.

- 사용자 자신에 대한 것(언어·스타일·습관) → `~/.claude/CLAUDE.md`(글로벌).
- 프로젝트 핵심 정보(구조·상태·워크플로우) → 프로젝트 `CLAUDE.md`.
- 특정 폴더/작업에서만 필요(트러블슈팅·상세 규칙) → `.claude/rules/주제.md`.
- 개인용·git 제외(로컬 경로·개인 설정) → `CLAUDE.local.md`.

## 6. 글로벌 설정 변경 프로토콜

- `~/.claude/` 아래 파일(CLAUDE.md·settings.json·skills 등)을 고칠 때: ① 원본 저장소를 `git pull`로 최신화 → ② `dotfiles/`를 수정 → ③ `~/.claude/`에 복사(즉시 반영) → ④ 다른 PC 공유가 필요하면 `python sync.py` 실행.
- `~/.claude/`만 직접 고치면 다음 sync에서 덮어써져 날아간다.
